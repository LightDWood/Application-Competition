# 需求沟通功能测试计划

## 目标
模拟用户使用会话框进行需求沟通，测试功能的可用性和逻辑闭环性。

## 当前系统状态

### 已实现的核心服务
- `SessionService` - 会话管理（内存存储）
- `AgentService` - Agent生命周期管理
- `RouterService` - 消息路由和Provider回退
- `PromptBuilder` - 四层提示词合并
- `ConfigService` - 系统配置管理

### API路由
- `POST /api/v2/sessions` - 创建会话
- `GET /api/v2/sessions/:id` - 获取会话详情
- `GET /api/v2/sessions` - 获取会话列表
- `POST /api/v2/sessions/:id/chat` - 发送消息（SSE流式响应）
- `PUT /api/v2/sessions/:id` - 更新会话
- `DELETE /api/v2/sessions/:id` - 删除会话

### 前端集成
- `SessionList.vue` - 使用 `sessionApi.v2`
- `ChatWindow.vue` - 使用 `sessionApi.v2.chat()`

## 测试场景

### 场景1：创建会话
**步骤**：
1. 创建新会话
2. 验证会话ID生成
3. 验证会话列表显示新会话

### 场景2：发送消息并接收回复
**步骤**：
1. 在会话中发送"我想开发一个电商网站"
2. 验证消息保存到历史
3. 验证流式响应（检查`chunk`事件）
4. 验证回复保存到历史
5. 验证`complete`事件触发消息刷新

### 场景3：多轮对话
**步骤**：
1. 继续发送"需要用户登录功能"
2. 验证历史消息累积
3. 验证上下文连贯性

### 场景4：会话持久化
**步骤**：
1. 刷新页面
2. 验证会话历史加载
3. 验证可以继续对话

### 场景5：错误处理
**步骤**：
1. API Key无效时，验证错误消息显示
2. 验证fallback机制
3. 验证Session不存在时的404处理

## 测试执行

### 前置条件
1. 确保后端服务器运行在 http://localhost:3000
2. 确保前端运行在 http://localhost:5180
3. 确保 providers.yaml 中配置了有效的 API Key

### 执行方式
1. 手动测试 - 通过浏览器访问UI进行测试
2. API测试 - 使用curl或Postman测试API
