# 修复数据库索引问题计划

## 问题描述

服务器启动时出现数据库错误：
```
Error: Too many keys specified; max 64 keys allowed
sql: ALTER TABLE `users` CHANGE `username` `username` VARCHAR(50) NOT NULL UNIQUE;
```

**根本原因**：
- Sequelize 使用 `{ alter: true }` 模式同步数据库表
- 每个 `unique: true` 字段都会创建唯一索引
- 多次执行 alter 操作导致索引累积超过 MySQL 限制（64 个）

## 解决方案

### 方案 A：禁用 alter 模式（推荐）

**优点**：
- 简单快速，立即解决问题
- 避免不必要的表结构修改
- 性能更好

**缺点**：
- 需要手动管理数据库迁移
- 新增字段需要手动添加

**实施步骤**：
1. 修改 `app.js`，将 `alter: true` 改为 `alter: false`
2. 首次启动时使用 `force: false`（不删除数据）
3. 如需修改表结构，手动执行 SQL 或使用迁移脚本

### 方案 B：手动清理索引

**优点**：
- 保留 alter 模式的便利性
- 清理冗余索引

**缺点**：
- 需要数据库操作权限
- 可能影响现有数据

**实施步骤**：
1. 连接到 MySQL 数据库
2. 查看 users 表的所有索引
3. 删除不必要的索引
4. 保留主键和必要的唯一索引

### 方案 C：使用数据库迁移（长期方案）

**优点**：
- 专业的数据库变更管理
- 可追溯、可回滚
- 适合团队协作

**缺点**：
- 需要引入迁移工具
- 学习成本

**实施步骤**：
1. 安装 sequelize-cli
2. 创建迁移脚本
3. 禁用自动同步
4. 使用迁移管理表结构变更

## 推荐方案

**立即执行**：方案 A（禁用 alter 模式）
**长期规划**：方案 C（数据库迁移）

## 实施步骤（方案 A）

### Step 1: 修改 app.js
- 将 `await db.sequelize.sync({ alter: true })` 改为 `await db.sequelize.sync()` 
- 或者添加配置选项控制

### Step 2: 清理现有索引（可选）
- 提供 SQL 脚本清理 users 表的冗余索引
- 保留必要的唯一约束

### Step 3: 测试验证
- 重启服务器
- 验证数据库连接正常
- 验证 CRUD 操作正常

### Step 4: 文档更新
- 记录数据库管理流程
- 说明如何手动添加字段/索引

## 验收标准

- [ ] 服务器成功启动，无索引错误
- [ ] 用户登录/注册功能正常
- [ ] 数据库 CRUD 操作正常
- [ ] 性能无明显下降

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 表结构不同步 | 低 | 中 | 提供手动同步脚本 |
| 新增字段失败 | 中 | 中 | 文档说明手动添加流程 |
| 数据丢失 | 极低 | 高 | 不使用 force: true，仅 alter: false |
